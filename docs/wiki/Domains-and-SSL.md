# Domains & SSL

Expose an app to the internet with automatic HTTPS — one managed Caddy
container, zero config files to hand-edit.

## How it works

- The dashboard installs a single `pinqops-caddy` container (image
  `caddy:2-alpine`) that publishes ports **80** and **443** and joins the
  shared `pinqops-apps` network.
- Every route is `domain → container:port`. Caddy reaches the target by
  container DNS on the shared network and terminates TLS with **automatic
  Let's Encrypt certificates** (issued and renewed by Caddy).
- Certificates live in the named volumes `pinqops-caddy-data` /
  `pinqops-caddy-config`, so they survive container recreates and even a proxy
  uninstall/reinstall.
- The Caddyfile is generated from strictly validated fields (domain grammar,
  container-name charset, port range) — no free-text ever reaches the config.

## Using it

1. Point your domain's **DNS A record** at the server's public IP.
2. Dashboard → **Domains & SSL** → *Install proxy* (fails clearly if another
   process already holds 80/443).
3. Set the ACME email (Let's Encrypt expiry notices).
4. Add a route: domain, target container (dropdown), container port.

The first HTTPS request may take a few seconds while the certificate is
issued.

## Routing your deployed app

Caddy can only reach containers on the `pinqops-apps` network:

- **New setups** — the compose template generated by the dashboard already
  joins `pinqops-apps`; nothing to do.
- **Catalog apps** — always on the shared network; routable out of the box.
- **Existing compose files** — adding a route offers a one-click *connect*;
  note that a compose recreate drops a manual connection. The durable fix is
  adding the network to your compose file:

  ```yaml
  services:
    app:
      networks: [default, pinqops-apps]
  networks:
    pinqops-apps:
      external: true
  ```

Removing the proxy takes every route offline but keeps the certificate
volumes.
